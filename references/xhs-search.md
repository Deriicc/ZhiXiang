# 小红书信息检索（XHS Search）

当用户想了解「某所具体学校 / 具体专业 / 顶岗实习」的**真实学生视角**时，可以检索小红书作为官方信息的补充。

本文件规定三件事：
1. **怎么查**（检测环境 → 降级策略）
2. **查什么**（触发条件与查询构造）
3. **怎么告诉用户**（透明度要求）

实际读到的笔记内容，**必须**再走一遍 `references/xhs-content-evaluation.md` 的过滤与再包装规则后才呈现给用户。

---

## 一、检索路径优先级（硬约束）

**每次需要小红书数据时，按以下顺序尝试，前一种失败才降级到下一种：**

### 路径 A：本地 `xiaohongshu-mcp`（优先）

先检测用户环境里是否存在 xiaohongshu 相关的 MCP server。

**检测方法（任选其一）：**

1. 用 `Glob` 扫 MCP 描述符目录：
   ```
   /home/derick/.cursor/projects/home-derick-ZhiXiang/mcps/**/xiaohongshu*
   ```
   若返回任何文件，说明用户已配置；进一步用 `Glob` 在同目录下找工具描述符（通常是 `tools/search*.json` 或 `tools/feeds*.json`）。

2. 或用 `Glob` 直接搜：
   ```
   /home/derick/.cursor/projects/**/xiaohongshu*/tools/*.json
   ```

**确认工具存在后：**

- 先 `Read` 工具描述符 JSON 确认参数 schema（必做——MCP 工具参数因版本不同会有差异）。
- 用 `CallMcpTool` 调用对应的搜索工具，典型参数：
  ```json
  {
    "keyword": "{学校名或专业名}",
    "filters": {
      "sort_by": "综合",
      "note_type": "不限",
      "publish_time": "一周内",
      "search_scope": "不限",
      "location": "不限"
    }
  }
  ```
- `publish_time` 根据场景选择：
  - **择校实时感受**：`一周内` 或 `一个月内`
  - **专业长期体验**：`半年内`
  - **升学经验**：`半年内`

**登录态失效的情况**：若返回 `is_logged_in=false` 或类似的未登录错误，**不要**尝试任何"帮用户重新登录"的操作——告知用户本地 MCP 登录过期，降级到路径 B。

### 路径 B：通用网页搜索降级

当路径 A 不可用（未检测到 MCP / MCP 登录失效 / MCP 报错）时：

- 用 `WebSearch` 工具执行：
  ```
  site:xiaohongshu.com {学校名或专业名} {限定词，如 宿舍/真实/毕业/实习}
  ```
- **注意**：小红书笔记在通用搜索引擎里索引率不高，覆盖度远低于路径 A。命中结果**可能很少甚至为零**，这是预期行为，不要为了凑数硬编。

### 路径 C：两者都失败时

如果路径 A 和 B 都没拿到可用数据：
- **不要**编造小红书上的内容。
- 直接告知用户「这一轮没有取到小红书数据」（见下文透明度要求）。
- 改向用户建议：「你可以自己去小红书搜 `{关键词}`，挑 3-5 条像真实学生写的，发给我一起看」——这是 SKILL.md 里「帮助用户区分事实/评价/偏见」原则的体现，不是失败托词。

## 二、何时应该用小红书

### ✅ 适合触发

- 用户问**具体学校**的宿舍、食堂、氛围、管理风格、上课日常
- 用户问**具体专业**的实际学习内容、实习体验、对口就业方向
- 用户想了解**顶岗实习**的真实工资、工时、学校抽成情况
- 用户问「XX 学校到底怎么样 / 真实情况」类问题

### ⛔ 不要触发

- **情绪类模式**（偏见与社会压力、危机支持）——小红书内容极易加重焦虑，绝不自动搜。
- 升学政策/分数线/录取规则——这些必须走官方来源（教育厅/考试院），小红书信息既过期又不准。
- 用户已处于明显自我怀疑或羞耻状态——优先回应情绪，不要突然抛一堆"别人都考上了"的小红书帖。
- 用户没问具体学校/专业，只是泛泛谈"职校"——搜了也是一堆骂战贴和营销贴。

### 🤔 先问用户再触发

- 如果属于"合规但高风险"——例如用户问"XX 学校是不是真的很差"——先询问用户：「要不要我去小红书上看看在校生和毕业生怎么说？小红书上信息比较杂，我会帮你过滤。」**征得同意再查**。

## 三、透明度要求（硬约束）

**任何一次使用小红书信息的回答，必须在结尾显式标注使用的路径。** 不要假装数据来自模型记忆。

在呈现小红书分析段落前，加入一个**标题行**（让用户一眼看到「这部分是小红书内容」）；在段落结尾加入**数据来源标注**，格式如下：

### 路径 A 成功时

```
### 📱 小红书上的学生声音

（……过滤并再包装后的内容……）

---
ℹ️ 数据来源：通过本地 xiaohongshu-mcp 检索到 N 条笔记，按相关性/时间筛选后保留 M 条呈现给你。
时间范围：{一周内 / 一个月内 / 半年内}。
```

### 路径 B 降级时

```
### 📱 小红书上的学生声音（网页搜索模式）

（……过滤并再包装后的内容……）

---
ℹ️ 数据来源：本地未检测到 xiaohongshu-mcp，改用通用网页搜索 `site:xiaohongshu.com`，
命中 N 条结果。这种方式覆盖度有限（通常只能拿到被搜索引擎收录的一部分），
如果想拿到更完整的视角，可以自己去小红书 App 搜「{关键词}」看一下。
```

### 路径 C 全部失败时

```
### 📱 关于小红书

这一轮我没能从小红书取到数据（本地 MCP 未配置，网页搜索也没返回相关结果）。

建议你自己去小红书搜「{具体关键词}」，挑 3-5 条看起来像真实在校生或毕业生写的
（避开明显营销账号、投稿号、剪辑号），把内容发给我，我帮你一起看哪些是事实、
哪些是个人情绪、哪些是偏见。
```

## 四、查询构造技巧

小红书的搜索相关性很依赖关键词构造。**不要**直接把用户原话丢进去。

### 择校场景

- ❌ 差：`"XX 职业学院怎么样"`
- ✅ 好：`"XX 职业学院 宿舍"`、`"XX 职业学院 真实"`、`"XX 职业学院 毕业"`、`"XX 职业学院 就读体验"`
- 建议**分两到三次查询**，每次换一个角度（宿舍/日常/毕业去向），再汇总。

### 专业场景

- ❌ 差：`"护理专业好不好"`
- ✅ 好：`"护理专业 实习"`、`"护理中职 日常"`、`"护理大专 毕业"`

### 实习场景

- ❌ 差：`"顶岗实习"`（全国性话题，噪音极大）
- ✅ 好：`"顶岗实习 工资"`、`"{行业}顶岗实习 真实"`、`"电子厂实习 学生"`

### 多次查询的上限

- **路径 A**：最多 3 次查询，每次 `limit=15` 以内，合计不超过 45 条候选。
- **路径 B**：最多 2 次 `WebSearch`。
- **任何一次 session 内不重复查同样的关键词。**

## 五、错误处理与重试

- MCP 单次超时（>20 秒）→ 不重试，直接降级到路径 B。
- MCP 明确返回"未登录" → 不重试，告诉用户登录过期，降级到路径 B。
- `WebSearch` 返回空 → 不重试（空就是空，不是失败）；直接进入透明度提示中的"命中 0 条"分支。
- 任何 rate-limit / 5xx → **不**尝试 exponential backoff（zhixiang 是对话型 skill，用户正在等你，不要吊着）。直接降级或如实告知。

## 六、隐私与合规边界

- **不要**让用户提供他们自己的小红书账号、cookie、手机号——MCP 是**用户主动配置**的东西，我们不诱导配置。
- **不要**复述笔记里的**可识别个人信息**（博主真名、学号、具体寝室号、完整全家信息等）——即便小红书笔记里有。
- **不要**在回答里嵌入小红书笔记的完整长 URL——用"在校生视角"、"一位毕业生在小红书写道"这类引用方式即可。若用户明确要原文链接，再给。
- **小红书笔记截图/正文里如果出现其他未成年人的面孔或真实姓名**：不引用、不描述、不复述。
