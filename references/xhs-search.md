# 小红书信息检索（XHS Search）

小红书检索是本 skill 的**可选加分项**——需要用户本地配置 `xiaohongshu-mcp` 服务。因此**仅当用户消息里显式出现"小红书 / 小红薯 / xhs / 红书"字眼时**，才读取并遵循本文件；其他情况一律走普通 WebSearch + 官方来源（按 `references/sources-and-evidence.md`）。

> **阅读本文件前请先确认**：用户是否真的显式提了"小红书"？没有 → 立即关闭本文件，回到常规检索流程。有 → 继续按第二节判定是否进入硬红线排除范围；通过后再按第一节走检索。

本文件规定三件事：
1. **怎么查**（检测环境 → 降级策略）
2. **什么时候不查**（显式触发 + 硬红线）
3. **怎么告诉用户**（透明度要求）

实际读到的笔记内容，**必须**再走一遍 `references/xhs-content-evaluation.md` 的过滤与再包装规则后才呈现给用户。

## 一、检索路径优先级（硬约束）

**每次需要小红书数据时，按以下顺序尝试，前一种失败才降级到下一种：**

### 路径 A：本地 `xiaohongshu-mcp`（优先）

先检测用户环境里是否存在 xiaohongshu 相关的 MCP server。

**检测方法（任选其一）：**

1. 用 `Glob` 扫 MCP 描述符目录：
   ```
   /home/derick/.cursor/projects/home-derick-ZhiXiang/mcps/**/xiaohongshu*
   ```
   若返回任何文件，说明用户已配置；进一步用 `Glob` 在同目录下找工具描述符（通常是 `tools/search*.json` 或 `tools/feeds*.json`）。

2. 或用 `Glob` 直接搜：
   ```
   /home/derick/.cursor/projects/**/xiaohongshu*/tools/*.json
   ```

**确认工具存在后：**

- 先 `Read` 工具描述符 JSON 确认参数 schema（必做——MCP 工具参数因版本不同会有差异）。
- 用 `CallMcpTool` 调用对应的搜索工具，典型参数：
  ```json
  {
    "keyword": "{核心识别词，具体构造见第四节}",
    "filters": {
      "sort_by": "综合",
      "note_type": "不限",
      "publish_time": "不限",
      "search_scope": "不限",
      "location": "不限"
    }
  }
  ```
- `keyword` 的构造是**成败关键**——错了会超时或 0 结果，详见第四节。默认方向：**学校查询只用简称（不加任何限定词）**。
- `publish_time` 默认 `不限`。冷门中职/职校笔记稀少，时间窗口过窄（一周内/一个月内）会把原本就不多的命中进一步过滤掉。只有用户明确问"最近情况"时才用 `一周内`。

**登录态失效的情况**：若返回 `is_logged_in=false` 或类似的未登录错误，**不要**尝试任何"帮用户重新登录"的操作——告知用户本地 MCP 登录过期，降级到路径 B。

### 路径 B：通用网页搜索降级

当路径 A 不可用（未检测到 MCP / MCP 登录失效 / MCP 报错）时：

- 用 `WebSearch` 工具执行（关键词规则与路径 A 一致，详见第四节）：
  ```
  site:xiaohongshu.com {核心识别词，如 振华外经 / 护理 中职 / 浦东 职校}
  ```
- **不要**在 `site:` 查询里叠加限定词（宿舍/真实/毕业等）——理由同路径 A：冷门主体笔记基数小，叠词必然 0 结果。
- **注意**：小红书笔记在通用搜索引擎里索引率不高，覆盖度远低于路径 A。命中结果**可能很少甚至为零**，这是预期行为，不要为了凑数硬编。

### 路径 C：两者都失败时

如果路径 A 和 B 都没拿到可用数据：
- **不要**编造小红书上的内容。
- 直接告知用户「这一轮没有取到小红书数据」（见下文透明度要求）。
- 改向用户建议：「你可以自己去小红书搜 `{关键词}`，挑 3-5 条像真实学生写的，发给我一起看」——这是 SKILL.md 里「帮助用户区分事实/评价/偏见」原则的体现，不是失败托词。

## 二、何时应该用小红书（显式触发）

小红书检索是本 skill 的**可选加分项**——需要用户本地配置并运行 `xiaohongshu-mcp` 服务，是有门槛的。因此**只在用户明确说他想用小红书时**才启用本文件的流程，其他情况一律走普通 WebSearch + 官方来源（按 `references/sources-and-evidence.md`）。

### ✅ 唯一的启用条件

**用户消息中必须出现显式的"小红书"字眼**。满足其一即可：

- 关键词：`小红书`、`小红薯`、`xhs`、`XHS`、`红书`
- 句式：「去小红书上看看」、「小红书上怎么说」、「搜一下小红书」、「用小红书查查」、「我听小红书上说 …… 是不是真的」

**没有这类字眼，即使用户问"宿舍/食堂/真实情况/口碑/学长学姐/就读体验/在读"等等，都不启用本文件的流程**——这类问题按普通信息检索处理（`references/sources-and-evidence.md`），用官方来源 + 通用网页搜索即可。

> **为什么收紧？** 之前按"触发词白名单"自动发起 MCP 调用，会在没配置这个服务的用户那里无意义地失败或超时，污染体验。现在改为显式触发后，没配置 MCP 的用户永远不会遇到这份规范；配置了的用户在真正想用时一句话就能唤起。

### ⛔ 即使用户说了"小红书"也不启用的硬红线

- **当前处于情绪 / 危机模式**（偏见与社会压力、危机支持）——小红书内容会加重焦虑。即使用户主动要求，**先询问是否真的想看**，并提示"小红书上比较乱，我会帮你过滤"，征得同意再查。
- **升学政策 / 分数线 / 录取规则 / 资助标准 / 实习管理规定**——这些小红书信息既过期又不准。可以一句话告诉用户"这块我不走小红书，上考试院/教育厅官网更靠谱"，然后按官方来源回答。
- **泛问**（没有具体学校 / 专业 / 行业主体，如"读职校怎么样"、"中职值不值得"）——搜回来全是骂战和营销，没价值。让用户先锚定具体对象再查。

## 三、透明度要求（硬约束）

**任何一次使用小红书信息的回答，必须在结尾显式标注使用的路径。** 不要假装数据来自模型记忆。

在呈现小红书分析段落前，加入一个**标题行**（让用户一眼看到「这部分是小红书内容」）；在段落结尾加入**数据来源标注**，格式如下：

```

时间范围：{不限 / 一周内 / 一个月内 / 半年内}（取 publish_time 实际值）。
```

### 路径 B 降级时

```
### 📱 小红书上的学生声音（网页搜索模式）

（……过滤并再包装后的内容……）

---
ℹ️ 数据来源：本地未检测到 xiaohongshu-mcp，改用通用网页搜索 `site:xiaohongshu.com`，
命中 N 条结果。这种方式覆盖度有限（通常只能拿到被搜索引擎收录的一部分），
如果想拿到更完整的视角，可以自己去小红书 App 搜「{关键词}」看一下。
```

### 路径 C 全部失败时

```

（避开明显营销账号、投稿号、剪辑号），把内容发给我，我帮你一起看哪些是事实、
哪些是个人情绪、哪些是偏见。
```

## 四、查询构造技巧

小红书的搜索本质是**模糊匹配 + 推荐**，不是结构化检索。**实测结论**：对冷门中职/职校，`{学校简称}` 能命中，但 `{学校简称} 宿舍`、`{学校简称} 真实`、`{学校简称} 就读体验`、`{学校简称} 校区` 这类"核心词 + 限定词"组合**几乎必然 0 结果**——因为这类组合要求同一篇笔记里同时出现两个词，而冷门主体的笔记基数太小，叠一个限定就被过滤光。更糟的情况是触发异常页，让上游 go-rod 无限等待点击元素，直到客户端 60 秒超时。

因此：**维度细化（宿舍 / 口碑 / 毕业去向 / 校风 etc.）不在查询层解决，在内容层解决**——首查核心词拿回笔记后，走 `references/xhs-content-evaluation.md` 做话题标注和提炼。

### 四条不可违反的原则

1. **首查只用核心识别词**（学校简称 / 专业+层次 / 地区+品类 / 行业+实习），**不叠加任何限定词**。
2. **关键词长度硬顶 ≤ 15 字**。学校全名超过 15 字 → 必须先简化到简称再查。
3. **keyword 字段里禁止出现**以下词（无论首查还是二次查）：`宿舍 / 校区 / 真实 / 就读 / 毕业 / 日常 / 体验 / 口碑 / 怎么样 / 好不好 / 值不值 / 靠谱`。这些词可以出现在 `xhs-content-evaluation.md` 的笔记分类标签里，但**不能**进 MCP 的 keyword 字段。
4. **fallback 方向只有"砍词"一条**。首查 0 结果 → 砍到更短的核心词再查 1 次；仍无 → 停查并告知用户。**不得**改叠限定词再试。

### 四类场景关键词模板（路径 A & B 通用）

| 场景 | 用户问题形态 | 关键词（终态，核心识别词） | 例子 |
|---|---|---|---|
| **A. 查具体学校**（已知校名） | "XX 学校怎么样 / 真实情况 / 宿舍如何" | **学校简称**（2-8 字） | `振华外经`、`上海商贸旅游`、`杭电信` |
| **B. 查具体专业**（已知专业） | "XX 专业值不值得读" | **专业名 + 层次**（压噪声） | `护理 中职`、`电子商务 高职`、`铁道信号 中专` |
| **C. 寻找学校**（泛问型） | "xx 地区有什么好的职校" | **地区 + 品类**（名词式，不整句） | `上海 中专`、`浦东 职校`、`深圳 护理 中专` |
| **D. 查实习行情** | "XX 行业顶岗实习怎样" | **行业 + 实习** | `电子厂 实习`、`护士 实习`、`物流 顶岗` |

> **注意**：表中"用户问题形态"里包含"宿舍 / 真实情况"这类意图词，不代表可以把这些词塞进 keyword。agent 必须识别用户意图，但 keyword 只写核心识别词。

### 关键词构造的硬规则

1. **长度硬顶 ≤ 15 字**。学校全名超过 15 字 → 必须先提取简称再查。
2. **keyword 字段禁用词清单**（同上）：`宿舍 / 校区 / 真实 / 就读 / 毕业 / 日常 / 体验 / 口碑 / 怎么样 / 好不好 / 值不值 / 靠谱`。
3. **禁止将自然语言问句整句作为 keyword**。用户说"上海浦东有什么好的职校啊"→ 提取为 `浦东 职校`，不要把整句丢入。
4. **地区词用行政区级别**（省/市/区），不自造缩写（❌"沪东" ✅"浦东"）。
5. **专业词避免只搜专业名**（"护理"会混进本科/成教噪声），必须加层次（`中职 / 中专 / 高职 / 大专`）。
6. **例外（热门主体不在此限）**：若主体本身是大学或热门品牌专业（如「上海交通大学 宿舍」「计算机 考研」），限定词叠加一般能命中。但**只要用户问的是中职 / 中专 / 职校 / 冷门高职**，一律按上面规则走。

### 学校简称提取启发式（当校名 > 15 字时）

按优先级尝试：

1. **官方简称**——若用户或上一步检索中出现官方简称，直接用。
2. **识别度最高的 2-6 字**——校名中去掉"上海 / 附属 / 浦东 / 职业技术 / 学校"等通用成分，保留最独特的部分：
   - 例：「上海第二工业大学附属浦东振华外经职业技术学校」→ **`振华外经`**
   - 例：「郑州市国防科技学校」→ **`郑州国防科技`** 或 **`郑州国防`**
   - 例：「深圳市第二职业技术学校」→ **`深圳二职`** 或 **`深二职`**
3. **实在提取不出独特词** → 跳过小红书检索，走路径 C 告知用户自查。**不要硬塞整长名**。

### 查询次数上限

- **路径 A（MCP）**：**单轮最多 2 次**。允许用途：
  - ✅ 首查 + 砍词 fallback（例：`振华外经` → `振华`）
  - ✅ 两个独立主体各查 1 次（用户同时问 A、B 两所学校或两个专业时）
  - ❌ 同一主体不同侧面（如 `振华外经` 之后再查 `振华外经 宿舍`）
  - ❌ 重复同一关键词
- **路径 B（WebSearch）**：**单轮最多 2 次**。关键词规则同上（`site:xiaohongshu.com 振华外经`，而不是 `site:xiaohongshu.com 振华外经 宿舍`）。
- **任一路径，单轮内不重复查同样的关键词**。

### Fallback 示例

**正面示例（首查命中，无需 fallback）**：

```
首查：振华外经          → 命中 N 条笔记（N ≥ 1）
停查：笔记交给 xhs-content-evaluation.md 做话题标注，
     在内容层筛出"宿舍 / 口碑 / 毕业去向"等维度。
```

**负面示例（首查 0 结果，砍词一次后停）**：

```
首查：振华外经          → 0 结果
再查：振华              → 仍无
停查：告知用户"本轮未在小红书查到该校相关内容"，给出自查建议。
```

**反模式（严格禁止）**：

```
❌ 首查：振华外经       → 命中 3 条
❌ 再查：振华外经 宿舍  → 0 结果（浪费预算，且可能触发异常页超时）
❌ 再查：振华外经 真实  → 0 结果
```

方向只有一条：**砍词，不是叠词**。同一主体的不同侧面一律走内容层（`references/xhs-content-evaluation.md`），不走查询层。

## 五、错误处理与重试

- MCP 单次超时（>20 秒）→ 不重试，直接降级到路径 B。
- MCP 明确返回"未登录" → 不重试，告诉用户登录过期，降级到路径 B。
- `WebSearch` 返回空 → 不重试（空就是空，不是失败）；直接进入透明度提示中的"命中 0 条"分支。
- 任何 rate-limit / 5xx → **不**尝试 exponential backoff（zhixiang 是对话型 skill，用户正在等你，不要吊着）。直接降级或如实告知。

## 六、隐私与合规边界

- **不要**让用户提供他们自己的小红书账号、cookie、手机号——MCP 是**用户主动配置**的东西，我们不诱导配置。
- **不要**复述笔记里的**可识别个人信息**（博主真名、学号、具体寝室号、完整全家信息等）——即便小红书笔记里有。
- **不要**在回答里嵌入小红书笔记的完整长 URL——用"在校生视角"、"一位毕业生在小红书写道"这类引用方式即可。若用户明确要原文链接，再给。
- **小红书笔记截图/正文里如果出现其他未成年人的面孔或真实姓名**：不引用、不描述、不复述。
